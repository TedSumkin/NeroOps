# План тестов для `test_health_analytics.py`

Документ описывает, какие тесты нужно написать для первой итерации аналитики
здоровья. Реализация тестов остаётся ручной: здесь нет готового кода тестов,
только план, ожидаемое поведение и рабочий порядок.

## Область задачи

Первая итерация ограничена контрактом `HA-001` из `CONTRACTS.md`.

Тестируемая функция:

```text
count_symptom_free_days(entries, from_date, to_date, timezone_name="Europe/Moscow") -> int
```

Файл тестов:

```text
tests/test_health_analytics.py
```

На этом этапе не нужно тестировать:

- `symptom_frequency()`;
- `build_health_period_metrics()`;
- сравнение периодов;
- ассоциации еды и симптомов;
- FastAPI;
- базу данных;
- frontend.

Тесты должны быть чистыми unit-тестами над обычным списком объектов `Entry`.

## Что проверить перед началом

1. Прочитать `CONTRACTS.md`, раздел `HA-001`.
2. Посмотреть `Entry` и `EntryType` в `backend/neroops/models.py`.
3. Посмотреть стиль существующих тестов в `tests/test_reports.py`.
4. Убедиться, что тесты будут создавать `Entry` напрямую, без базы данных.

## Подготовка тестового файла

В `tests/test_health_analytics.py` нужна минимальная инфраструктура:

- импорты `date`, `datetime`, `UTC`;
- импорт `pytest`;
- импорт `Entry` и `EntryType`;
- импорт `count_symptom_free_days`;
- небольшой helper для создания `Entry`.

Helper должен создавать объект `Entry` напрямую. Он не должен сохранять данные в
SQLite, обращаться к API или использовать фикстуры FastAPI.

## Рабочий порядок

Пиши тесты по одному.

После каждого нового теста запускай только файл с этими тестами:

```bash
./.venv/bin/pytest tests/test_health_analytics.py -q
```

Когда все тесты для `HA-001` написаны и проходят, запускай весь набор:

```bash
./.venv/bin/pytest -q
```

После реализации проверь Ruff:

```bash
./.venv/bin/ruff check backend tests
```

## Рекомендуемый TDD-порядок

Перед написанием тестов можно создать пустой модуль
`backend/neroops/services/health_analytics.py` с функцией правильной сигнатуры,
которая пока явно не реализована.

Это полезно, потому что тесты будут падать из-за отсутствующего поведения, а не
из-за `ImportError`.

## Обязательные тестовые сценарии

### 1. Пустой список записей

Сценарий:

- `entries` пустой;
- период состоит из нескольких дней;
- `from_date` и `to_date` включаются в расчёт.

Ожидание:

- функция возвращает количество всех дней периода.

Что проверяет:

- пустой ввод;
- включительный расчёт периода;
- базовую формулу `period_days = (to_date - from_date).days + 1`.

Пример смысла:

- период с 10 июня по 12 июня содержит 3 дня;
- симптомов нет;
- результат должен быть `3`.

### 2. Два симптома в один день

Сценарий:

- в периоде есть две записи `EntryType.symptom`;
- обе записи попадают на одну локальную календарную дату;
- период содержит больше одного дня.

Ожидание:

- этот день считается симптомным только один раз;
- результат уменьшается на `1`, а не на количество записей симптомов.

Что проверяет:

- функция считает дни с симптомами, а не количество строк;
- для `HA-001` `payload["count"]` не влияет на то, считается ли день симптомным.

Пример смысла:

- период содержит 3 дня;
- два симптома записаны в первый день;
- дней без симптомов должно остаться `2`.

### 3. Симптом вне периода

Сценарий:

- период валидный;
- есть симптом до начала периода или после конца периода.

Ожидание:

- симптом вне периода игнорируется;
- результат равен количеству всех дней периода, если внутри периода симптомов нет.

Что проверяет:

- фильтрацию записей по локальной дате;
- корректное игнорирование лишних данных.

Пример смысла:

- период с 10 июня по 12 июня;
- симптом записан 9 июня или 13 июня;
- результат должен быть `3`.

### 4. Timezone около полуночи

Сценарий:

- запись симптома имеет время `2026-06-14 21:30 UTC`;
- timezone для расчёта: `Europe/Moscow`.

Ожидание:

- в Москве это уже `2026-06-15 00:30`;
- симптом должен относиться к локальной дате `2026-06-15`.

Что проверяет:

- перевод `occurred_at` в переданный timezone перед извлечением даты;
- важную границу около полуночи.

Пример смысла:

- период состоит только из 15 июня;
- симптом записан как `2026-06-14 21:30 UTC`;
- результат должен быть `0`, потому что 15 июня является симптомным днём.

### 5. Неверный диапазон дат

Сценарий:

- `to_date` раньше `from_date`.

Ожидание:

- функция выбрасывает `ValueError`.

Что проверяет:

- контракт периода;
- отсутствие попыток молча исправлять неправильные входные данные.

Для этого сценария можно передать пустой список `entries`: ошибка должна быть
связана именно с диапазоном дат.

### 6. Naive datetime

Сценарий:

- в `Entry.occurred_at` передан `datetime` без `tzinfo`;
- диапазон дат при этом валидный.

Ожидание:

- функция выбрасывает `ValueError`.

Что проверяет:

- контракт входных данных;
- функция не должна угадывать timezone для naive datetime.

Важно, чтобы диапазон был корректным. Иначе тест может проверить не ту ошибку.

## Имена тестов

Названия тестов должны описывать поведение, а не детали реализации.

Хороший стиль:

- `test_returns_all_days_for_empty_entries`;
- `test_counts_multiple_symptoms_on_same_day_once`;
- `test_ignores_symptoms_outside_period`;
- `test_uses_requested_timezone_for_local_day`;
- `test_raises_for_invalid_date_range`;
- `test_raises_for_naive_datetime`.

## Структура каждого теста

Держи одинаковую схему:

1. Arrange: подготовить даты и список `Entry`.
2. Act: вызвать `count_symptom_free_days()`.
3. Assert: проверить результат или ожидаемое исключение.

Не смешивай несколько разных причин падения в одном тесте.

## Что не проверять

Не нужно проверять внутреннюю реализацию:

- какие множества использует функция;
- есть ли приватные helper-функции;
- как именно устроен цикл;
- сортируются ли записи внутри функции.

Тесты должны проверять только публичное поведение из контракта.

## Минимальный критерий готовности

Задача по тестам готова, когда:

- все 6 обязательных сценариев покрыты;
- тесты не используют БД, API и файловую систему;
- тесты читаются как спецификация поведения;
- `./.venv/bin/pytest tests/test_health_analytics.py -q` проходит;
- после реализации `./.venv/bin/pytest -q` проходит;
- после реализации `./.venv/bin/ruff check backend tests` проходит.
