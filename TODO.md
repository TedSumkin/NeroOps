# NeroOps: мой план разработки на Python

Проект наполовину навайбкожен. Этот файл описывает ту часть NeroOps, которую пишет владелец проекта. Детальные
сигнатуры и правила поведения находятся в [CONTRACTS.md](CONTRACTS.md).

## Backlog
Сделать интерфейс менее вырвиглазным. М.б. в перспективе заказать у кого-нибудь нормальное оформление.


## Что делать первым

Первая итерация ограничена одним контрактом: `HA-001`.

1. Создать `tests/test_health_analytics.py`.
2. Написать тесты `count_symptom_free_days()` из раздела `HA-001`.
3. Создать `backend/neroops/services/health_analytics.py`.
4. Реализовать функцию до прохождения этих тестов.
5. Не начинать `HA-002`, пока код `HA-001` не прошёл review.

Так первая задача остаётся небольшой: даты, timezone, множества и тесты. В ней
нет SQL, FastAPI, frontend или статистических допущений.

## Моя зона ответственности

Я пишу и изучаю:

- чистую Python-аналитику над `list[Entry]`;
- Pydantic-модели результатов аналитики;
- unit-тесты этих функций;
- позднее — более сложный анализ временных рядов, когда накопятся данные.

На первом этапе я **не трогаю**:

- SQLAlchemy-запросы и миграции;
- FastAPI-эндпоинты;
- React/TypeScript и интерфейс iPhone;
- IndexedDB, service worker и офлайн-синхронизацию;
- Tailscale, launchd, Docker и работу с файлами.

Эти границы нужны не потому, что перечисленные части недоступны для изучения, а 
чтобы первая работа была обычным понятным Python без пяти новых технологий
одновременно.

## Архитектура моей части

Поток данных:

```text
SQLite
  -> SQLAlchemy получает list[Entry]       # готово, не моя задача
  -> health_analytics.py                   # моя чистая Python-логика
  -> Pydantic-модели результата            # моя задача
  -> FastAPI превращает результат в JSON   # подключит Codex
  -> PWA показывает карточки и графики     # подключит Codex
```

Основные файлы:

```text
backend/neroops/services/health_analytics.py  # создать, основная реализация
backend/neroops/schemas.py                    # добавить модели результата
tests/test_health_analytics.py                # создать, unit-тесты
```

Функции аналитики не должны:

- открывать БД или выполнять SQL;
- читать файлы или переменные окружения;
- обращаться к сети;
- изменять переданный список или объекты `Entry`;
- зависеть от FastAPI;
- делать медицинские или причинные выводы.

## Definition Of Done для каждой задачи

Задача считается готовой, если:

1. Сигнатура и поведение соответствуют `CONTRACTS.md`.
2. Есть тест нормального сценария, пустого ввода и ключевой границы.
3. Функция не мутирует аргументы и не выполняет I/O.
4. Все даты обрабатываются в переданном часовом поясе.
5. `.venv/bin/ruff check backend tests` проходит.
6. `.venv/bin/pytest -q` проходит.
7. Названия и docstring описывают вычисление, а не медицинский вывод.

## Что изучать по мере необходимости

Для этапов 1–4 достаточно:

- `collections.Counter` и `defaultdict`;
- `datetime`, `date`, `timedelta`;
- `zoneinfo.ZoneInfo`;
- `statistics.mean` и `median`;
- сортировка с `key=`;
- Pydantic `BaseModel` и `Field`;
- `pytest.mark.parametrize`.

SQL, pandas, NumPy и sklearn для этих этапов не нужны.

## После накопления данных

Не начинать ML только потому, что это знакомая область. Сначала нужны достаточно
полные наблюдения и явная оценка пропусков.

К возможным следующим задачам возвращаемся не раньше, чем накопится примерно
90 дней данных:

- недельные тренды симптомов и веса;
- анализ пропусков и регулярности ведения дневника;
- EWMA или robust rolling baseline для отклонений;
- лаговые связи между кормлением, активностью и симптомами;
- экспорт исследовательской таблицы для notebook.

Даже на этом этапе результаты формулируются как наблюдаемые ассоциации, а не как
диагноз или причинный эффект.
